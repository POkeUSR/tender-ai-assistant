# Tender AI Assistant — Руководство по приложению

Tender AI Assistant — это интеллектуальная система для анализа тендерной документации с использованием технологий искусственного интеллекта. Приложение помогает компаниям автоматизировать процесс подготовки к участию в государственных закупках, обеспечивая глубокий анализ рисков, проверку требований и генерацию заявок.

---

## 1. Обзор проекта

### Назначение системы

Tender AI Assistant представляет собой комплексное решение для автоматизации анализа тендерной документации. Система использует современные технологии машинного обучения и обработки естественного языка для извлечения структурированной информации из неструктурированных документов, оценки рисков и помощи в подготовке заявок на участие в закупках.

### Ключевые возможности

Система предоставляет широкий спектр функциональных возможностей, включая загрузку и обработку документов в различных форматах, таких как PDF, DOCX, DOC и TXT. После загрузки документа приложение выполняет автоматическое индексирование содержимого с использованием векторного хранилища FAISS, что обеспечивает быстрый семантический поиск по документу. Пользователи могут взаимодействовать с системой через интуитивный чат, задавая вопросы о содержимом тендерной документации и получая мгновенные ответы. Кроме того, приложение выполняет полный анализ тендера, включающий извлечение ключевой информации о закупке, сроках, требованиях и финансовых условиях. Особую ценность представляет функция анализа рисков, которая выявляет потенциальные риски дисквалификации, финансовые, технические, правовые и операционные риски, предоставляя рекомендации по их снижению.

### Целевая аудитория

Система предназначена для компаний, участвующих в государственных закупках, тендерных специалистов, закупочных отделов, а также консалтинговых компаний, оказывающих услуги по подготовке тендерной документации.

## 2. Архитектура приложения

### Общая архитектура

Приложение построено на основе современной трёхуровневой архитектуры, включающей серверную часть на Python с использованием FastAPI, клиентскую часть на Vue.js и систему RAG (Retrieval Augmented Generation) для работы с документами. Такое разделение обеспечивает масштабируемость, удобство сопровождения и возможность независимого развития каждого компонента.

### Технологический стек

**Backend** реализован на Python с использованием следующих ключевых библиотек: FastAPI в качестве веб-фреймворка, SQLAlchemy 2.0 с asyncpg для работы с базой данных PostgreSQL в асинхронном режиме, python-jose для работы с JWT токенами, bcrypt для хэширования паролей, а также langchain, langchain-openai и langchain-community для интеграции с языковыми моделями и работы с векторными хранилищами. FAISS используется для семантического поиска по документам, а PyMuPDF (fitz) и python-docx — для извлечения текста из PDF и DOCX файлов соответственно.

**Frontend** построен на Vue.js 3 с использованием Composition API, что обеспечивает модульность и переиспользуемость компонентов. Маршрутизация осуществляется через vue-router, стилизация выполнена с использованием Tailwind CSS, а для работы с Server-Sent Events (SSE) реализован специальный composable. Сборка проекта выполняется с помощью Vite.

**База данных** использует PostgreSQL в качестве СУБД с асинхронным драйвером asyncpg для обеспечения высокой производительности при работе с большим количеством запросов.

### Паттерны проектирования

На бэкенде применяется многослойная архитектура с чётким разделением на уровни. API endpoints обрабатывают входящие запросы и валидацию, бизнес-логика вынесена в services, работа с данными осуществляется через models, а схемы данных определены в schemas. Такое разделение обеспечивает слабую связанность компонентов и упрощает тестирование каждого слоя независимо от других.

### Файл backend/app/main.py

Этот файл является точкой входа в приложение FastAPI. Здесь выполняется инициализация приложения, настройка CORS middleware, подключение роутеров API и определение lifespan-менеджера для управления жизненным циклом приложения. При запуске приложения автоматически создаются таблицы в базе данных через функцию `init_db()`.

### Модуль backend/app/core/

**config.py** содержит класс настроек `Settings`, который загружает переменные окружения из `.env` файла. Здесь определены параметры подключения к базе данных, API ключи OpenAI, настройки JWT токенов, параметры LLM модели (gpt-4o-mini), модель эмбеддингов (text-embedding-3-large), размер чанков для разбиения текста (chunk_size=1200, chunk_overlap=200), а также CORS настройки для разрешённых источников.

**database.py** настраивает асинхронное подключение к PostgreSQL с использованием SQLAlchemy 2.0. Файл экспортирует асинхронный движок `engine`, фабрику сессий `AsyncSessionLocal`, базовый класс моделей `Base` и функцию `get_db()` для внедрения зависимостей в эндпоинты. Также содержит функции `init_db()` для создания таблиц и `drop_tables()` для их удаления.

**file_handler.py** отвечает за загрузку и обработку файлов различных форматов. Поддерживаются PDF (через PyMuPDF), DOCX (через python-docx), DOC (через antiword на Linux/Mac или win32com на Windows) и TXT. Основные функции: `load_file()` для загрузки одного файла, `load_files()` для загрузки нескольких файлов с объединением текста, а также словарь `LOADERS` с загрузчиками для каждого формата.

**prompt.py** содержит шаблоны промптов для AI анализа. Основной промпт `tender_prompt` использует библиотеку LangChain для формирования запросов к языковой модели. Также определены константы `ANALYZE_QUESTION` и `RISKS_QUESTION` с текстами запросов для полного анализа тендера и анализа рисков соответственно.

### Модуль backend/app/models/

**models.py** определяет SQLAlchemy модели для базы данных с использованием DeclarativeBase. Модель `Tender` представляет загруженный тендерный документ с полями: id, filename (имя файла), raw_text (сырой текст), status (статус обработки), created_at и updated_at. Модель `Analysis` хранит результаты AI анализа с полями: id, tender_id (связь с тендером), summary (краткое описание), risks (словарь рисков в JSON формате). Модель `Score`保存 оценки привлекательности тендера с полями: id, tender_id, value (оценка 0-100), reason (обоснование). Модель `Proposal` хранит сгенерированные заявки с полями: id, tender_id, text (текст заявки).

### Модуль backend/app/schemas/

**tender.py** содержит Pydantic схемы для валидации запросов и ответов API. Включает схемы для загрузки файлов (UploadResponse), проверки статуса (StatusResponse), чата (ChatRequest), документов (DocumentResponse, DocumentListResponse, DeleteResponse), аутентификации (UserCreate, UserResponse, Token, TokenRefreshRequest) и общие ответы (ErrorResponse, MessageResponse, ClearCacheResponse).

### Модуль backend/app/services/

**ai_service.py** реализует интеграцию с OpenAI API для выполнения различных задач анализа. Класс `AIService` предоставляет методы: `extract_data()` для извлечения структурированных данных из тендера (название, бюджет, сроки, требования), `analyze_risks()` для анализа рисков с категоризацией по типам, `get_score()` для расчёта оценки привлекательности тендера по шкале 0-100, и `generate_proposal()` для генерации текста заявки на участие. Все методы используют модель gpt-4o-mini и возвращают структурированные данные через Pydantic модели.

**tender_service.py** содержит бизнес-логику для работы с тендерами. Класс `TenderService` инкапсулирует все операции: загрузку файлов через `upload_files()`, чат с документом через `chat()`, полный анализ через `analyze()`, анализ рисков через `analyze_risks()` и получение статуса через `get_status()`. Внутренние методы `_split_text()`, `_create_vectorstore()` и `_stream_llm()` обеспечивают разбиение текста на чанки, создание FAISS векторного хранилища и потоковую генерацию ответов от LLM соответственно.

### Модуль backend/app/api/

**endpoints/tenders.py** определяет REST API эндпоинты для работы с тендерами. Эндпоинт `POST /api/upload` принимает файлы и создаёт тендер с индексацией. Эндпоинт `GET /api/status` возвращает текущий статус загруженного документа. Эндпоинт `POST /api/chat` принимает вопрос и возвращает потоковый ответ через SSE. Эндпоинт `POST /api/analyze` запускает полный анализ тендера. Эндпоинт `POST /api/risks` выполняет анализ рисков. Все эндпоинты используют зависимость `get_db` для получения асинхронной сессии базы данных.

### Модуль backend/state.py

Этот модуль управляет глобальным состоянием приложения, в частности векторными хранилищами FAISS. Класс `VectorStoreManager` обеспечивает потокобезопасное хранение векторных хранилищ с поддержкой как устаревшего однопользовательского режима, так и нового многопользовательского режима. Функции `set_vectorstore()` и `get_vectorstore()` предоставляют простой API для работы с единственным хранилищем, а функции `set_user_vectorstore()` и `get_user_vectorstore()` позволяют управлять отдельными хранилищами для каждого пользователя и документа.

### Компонент App.vue

Корневой компонент приложения реализует основной макет с боковой панелью навигации и областью контента. Боковая панель отображает логотип Tender AI Assistant, индикатор статуса загруженного документа (готов/не загружен), навигационные ссылки на четыре страницы и информацию о технологии GPT-4o-mini в подвале. Компонент загружает статус приложения при монтировании и предоставляет его через Vue provide/inject для доступа во всех дочерних компонентах.

### Маршрутизация (router/index.js)

Vue Router настроен с четырьмя основными маршрутами. Маршрут `/` перенаправляет на `/upload`. Маршрут `/upload` отображает страницу загрузки документов. Маршрут `/chat` отображает страницу чата с тендером. Маршрут `/analyze` отображает страницу полного анализа тендера. Маршрут `/risks` отображает страницу анализа рисков.

### Страница UploadView.vue

Страница загрузки документов предоставляет пользовательский интерфейс для выбора и загрузки файлов тендерной документации. Поддерживается drag-and-drop загрузка файлов, выбор нескольких файлов через диалоговое окно, отображение списка выбранных файлов с иконками форматов и размерами, фильтрация по поддерживаемым форматам (PDF, DOCX, DOC, TXT), загрузка файлов на сервер с индикацией прогресса и отображение результата с информацией об успешной загрузке и количестве созданных фрагментов. После успешной загрузки пользователь может перейти в чат или на страницу анализа.

### Страница ChatView.vue

Страница чата позволяет пользователям задавать вопросы о загруженном тендерном документе и получать мгновенные ответы от AI. Интерфейс включает пустое состояние с предложенными вопросами для быстрого старта, отображение сообщений пользователя и AI в виде пузырьков чата, автоматический рендеринг Markdown в ответах AI, потоковую загрузку ответов с анимацией курсора, автоматическую прокрутку к новым сообщениям, очистку истории чата и предупреждение о необходимости загрузить документ.

### Страница AnalyzeView.vue

Страница полного анализа тендера отображает структурированную информацию о закупке, извлечённую из документа AI. Включает краткое описание закупки, ключевые сроки подачи заявки, перечень обязательных документов, требования к участникам, финансовые условия, основные риски дисквалификации, чек-лист подготовки заявки и оценку сложности тендера.

### Страница RisksView.vue

Страница анализа рисков представляет детальную оценку рисков участия в тендере. Риски структурированы по категориям: риски дисквалификации заявки, финансовые риски, технические риски, правовые риски и операционные риски. Для каждого риска указываются описание, уровень (высокий/средний/низкий) и рекомендации по снижению.

### API клиент (api/index.js)

Модуль API предоставляет функции для взаимодействия с бэкендом. Функция `getStatus()` получает текущий статус загруженного документа. Функция `uploadFiles(files)` загружает файлы на сервер через multipart/form-data. Функция `uploadPDF(file)` является устаревшей обёрткой для обратной совместимости. Все функции возвращают промисы и обрабатывают ошибки с преобразованием их в исключения JavaScript.

### Composable useSSE.js

Composable для работы с Server-Sent Events обеспечивает потоковую передачу данных от сервера. Позволяет получать потоковые ответы от AI в реальном времени без необходимости ручной обработки читателя потока.

## 5. Система RAG

### Концепция RAG

Система RAG (Retrieval Augmented Generation) объединяет семантический поиск по документам с генерацией текста языковой моделью. Такой подход позволяет AI работать с конкретным тендерным документом, а не полагаться только на свои внутренние знания, что обеспечивает высокую точность и релевантность ответов.

### Компоненты RAG системы

```
Тендерный документ
       │
       ▼
┌──────────────────┐
│  File Loader     │  (backend/app/core/file_handler.py)
│  - PDF           │
│  - DOCX          │
│  - DOC           │
│  - TXT           │
└────────┬─────────┘
         │ Текст
         ▼
┌──────────────────┐
│  Text Chunker   │  (tender_service.py)
│  - chunk_size   │  Разбиение на фрагменты
│  - chunk_overlap│  с перекрытием
└────────┬─────────┘
         │ Чанки
         ▼
┌──────────────────┐
│  Embeddings     │  (OpenAI text-embedding-3-large)
│  Generation      │  Создание векторных представлений
└────────┬─────────┘
         │ Векторы
         ▼
┌──────────────────┐
│  FAISS Index    │  Векторное хранилище
│  (Vector Store) │  Семантический поиск
└────────┬─────────┘
         │ Релевантные фрагменты
         ▼
┌──────────────────┐
│  Prompt Template │  (backend/app/core/prompt.py)
│  + Context       │  Формирование запроса к LLM
└────────┬─────────┘
         │ Промпт
         ▼
┌──────────────────┐
│  LLM (GPT-4o)   │  Генерация ответа
└────────┬─────────┘
         │ Ответ
         ▼
    Пользователь
```

### Процесс индексации

При загрузке тендерного документа выполняется следующая последовательность действий. Сначала файл обрабатывается соответствующим загрузчиком в зависимости от формата, и из него извлекается текст. Затем текст разбивается на фрагменты (chunks) размером 1200 символов с перекрытием в 200 символов с использованием `RecursiveCharacterTextSplitter` из LangChain. Далее каждый фрагмент преобразуется в векторное представление с помощью модели эмбеддингов `text-embedding-3-large` от OpenAI. Векторы сохраняются в FAISS индекс, который используется для быстрого семантического поиска.

### Процесс поиска и генерации

При обработке запроса пользователя система выполняет семантический поиск наиболее релевантных фрагментов документа с использованием FAISS. Найденные фрагменты вместе с вопросом пользователя подставляются в шаблон промпта, который направляется в языковую модель GPT-4o-mini. Модель генерирует ответ на основе контекста из документа, обеспечивая точность и релевантность информации. Ответ передаётся пользователю в потоковом режиме через Server-Sent Events для мгновенного отображения.

### Параметры RAG

Система настраивается через параметры в `backend/app/core/config.py`. Размер чанка (chunk_size) по умолчанию составляет 1200 символов, что обеспечивает баланс между контекстуальной полнотой и точностью поиска. Перекрытие чанков (chunk_overlap) установлено в 200 символов для сохранения контекста на границах фрагментов. Модель эмбеддингов используется `text-embedding-3-large` как наиболее современная и точная модель для создания векторных представлений текста.

## 6. База данных

### Обзор схемы базы данных

Приложение использует PostgreSQL для хранения структурированных данных о тендерах, анализах, оценках и сгенерированных заявках. Все операции с базой данных выполняются асинхронно с использованием SQLAlchemy 2.0 и драйвера asyncpg.

### Таблицы базы данных

**Таблица tenders** хранит информацию о загруженных тендерных документах. Поле id является первичным ключом, поле filename хранит имя загруженного файла, поле raw_text содержит полный текст документа в сыром виде, поле status отражает текущий статус обработки (uploaded, processed, error), поля created_at и updated_at хранят временные метки создания и обновления записи.

**Таблица analyses** сохраняет результаты AI анализа тендеров. Поле id является первичным ключом, поле tender_id является внешним ключом на таблицу tenders, поле summary содержит краткое описание тендера, поле risks хранит словарь рисков в JSON формате, поле created_at фиксирует время создания анализа.

**Таблица scores** сохраняет оценки привлекательности тендеров. Поле id является первичным ключом, поле tender_id является внешним ключом на таблицу tenders, поле value содержит числовую оценку от 0 до 100, поле reason хранит текстовое обоснование оценки, поле created_at фиксирует время создания записи.

**Таблица proposals** хранит сгенерированные заявки на участие в тендере. Поле id является первичным ключом, поле tender_id является внешним ключом на таблицу tenders, поле text содержит полный текст сгенерированной заявки, поле created_at фиксирует время создания заявки.

### Взаимосвязи между таблицами

Между таблицами установлены связи один-ко-многим через SQLAlchemy relationships. Один тендер (Tender) может иметь много анализов (Analysis), много оценок (Score) и много сгенерированных заявок (Proposal). При удалении тендера все связанные записи удаляются каскадно благодаря параметру cascade="all, delete-orphan".

### Асинхронные операции

Все операции с базой данных выполняются асинхронно для обеспечения высокой производительности. Функция `get_db()` является генератором, который создаёт новую сессию для каждого запроса и автоматически закрывает её после завершения. Подключение к базе данных настраивается с использованием пула соединений (pool_size=5, max_overflow=10) для эффективной обработки множественных запросов.

## 7. Аутентификация

### JWT-базовая аутентификация

Система аутентификации использует JSON Web Tokens (JWT) для безопасной авторизации пользователей. JWT токены обеспечивают stateless аутентификацию без необходимости хранения сессий на сервере, что упрощает горизонтальное масштабирование приложения.

### Компоненты системы аутентификации

**Файл backend/auth.py** содержит все функции для работы с аутентификацией. Функция `verify_password()` проверяет соответствие пароля хэшу с использованием bcrypt. Функция `get_password_hash()` создаёт хэш пароля с использованием bcrypt. Функция `create_access_token()` создаёт JWT токен доступа с временем жизни 30 минут. Функция `create_refresh_token()` создаёт JWT токен обновления с временем жизни 7 дней. Функция `decode_token()` декодирует и валидирует JWT токен. Функция `get_current_user()` является FastAPI зависимостью для получения текущего аутентифицированного пользователя. Функция `authenticate_user()` выполняет аутентификацию по email и паролю.

### Структура JWT токена

Токен доступа содержит следующие поля: sub (идентификатор пользователя), exp (время истечения), type (тип токена — access). Токен обновления содержит те же поля, но с типом refresh и более длительным временем жизни.

### OAuth2 схема

Система использует схему OAuth2PasswordBearer для извлечения токена из заголовка Authorization. При запросе защищённых ресурсов клиент должен включить заголовок вида "Bearer <token>". Если токен отсутствует или недействителен, сервер возвращает код 401 с сообщением об ошибке.

### Безопасность паролей

Пароли пользователей никогда не хранятся в открытом виде. При регистрации пароль хэшируется с использованием bcrypt с автоматической генерацией соли. При аутентификации введённый пароль хэшируется и сравнивается с сохранённым хэшем. Такой подход обеспечивает защиту от взлома базы данных и атак методом перебора.

## 8. API эндпоинты

### Обзор API

Backend предоставляет REST API по базовому пути `/api`. Все эндпоинты возвращают данные в формате JSON, за исключением конечных точек чата, анализа и рисков, которые используют Server-Sent Events (SSE) для потоковой передачи данных.

### Список эндпоинтов

**Эндпоинт GET /** является корневым эндпоинтом, возвращающим информацию о приложении. Ответ содержит message с названием приложения и версией API.

**Эндпоинт GET /health** используется для проверки здоровья приложения. Возвращает статус "healthy" для мониторинга доступности сервиса.

**Эндпоинт POST /api/upload** принимает файлы тендерных документов для загрузки и обработки. Тело запроса содержит поле files с массивом файлов в формате multipart/form-data. Поддерживаются форматы PDF, DOCX, DOC и TXT. При успешной загрузке возвращается статус "ok", массив имён файлов и количество фрагментов текста. При ошибке формата возвращается код 400 с описанием ошибки.

**Эндпоинт GET /api/status** возвращает текущий статус загруженного документа. Ответ содержит поле ready (логическое значение готовности), поле filename (имя загруженного файла) и поле chunks_count (количество фрагментов). Если документ не загружен, ready возвращается как false, а остальные поля как null.

**Эндпоинт POST /api/chat** принимает вопрос пользователя о тендере и возвращает потоковый ответ. Тело запроса содержит поле question с текстом вопроса. Ответ передаётся в формате SSE с токенами в поле token. Формат ответа: `data: {"token": "часть ответа"}\n\n`. Конец ответа обозначается `data: [DONE]\n\n`. Если документ не загружен, возвращается код 400.

**Эндпоинт POST /api/analyze** запускает полный анализ тендера и возвращает потоковый ответ. Структура ответа аналогична эндпоинту чата. Анализ включает краткое описание закупки, ключевые сроки, обязательные документы, требования к участнику, финансовые условия, основные риски, чек-лист подготовки заявки и оценку сложности.

**Эндпоинт POST /api/risks** выполняет детальный анализ рисков и возвращает потоковый ответ. Структура ответа аналогична предыдущим эндпоинтам. Риски категоризированы по типам: дисквалификация, финансовые, технические, правовые и операционные. Для каждого риска указываются описание, уровень и рекомендации.

### Обработка ошибок

Все эндпоинты возвращают соответствующие коды состояния HTTP. Код 200 indicates успешное выполнение. Код 400 указывает на ошибку валидации или неверный запрос. Код 401 означает ошибку аутентификации. Код 500 означает внутреннюю ошибку сервера. Тела ошибок содержат поле detail с описанием проблемы.

### Примеры использования

Пример загрузки файла с использованием curl: `curl -X POST -F "files=@tender.pdf" http://localhost:8000/api/upload`. Пример получения статуса: `curl http://localhost:8000/api/status`. Пример отправки вопроса: `curl -X POST -H "Content-Type: application/json" -d '{"question":"Какие сроки подачи заявки?"}' http://localhost:8000/api/chat`.

## 9. Запуск приложения

### Требования

Для запуска приложения необходимы Python 3.10+, Node.js 18+, PostgreSQL база данных и OpenAI API ключ.

### Настройка переменных окружения

Создайте файл `.env` в корневой директории проекта со следующими переменными: OPENAI_API_KEY с вашим API ключом OpenAI, DATABASE_URL с строкой подключения к PostgreSQL в формате postgresql+asyncpg://user:password@host:port/dbname?sslmode=require, SECRET_KEY со secret key для JWT токенов.

### Запуск бэкенда

Перейдите в директорию backend и установите зависимости: `pip install -r requirements.txt`. Запустите сервер: `uvicorn app.main:app --reload --host 0.0.0.0 --port 8000`. Бэкенд будет доступен по адресу http://localhost:8000.

### Запуск фронтенда

Перейдите в директорию frontend и установите зависимости: `npm install`. Запустите dev сервер: `npm run dev`. Фронтенд будет доступен по адресу http://localhost:5173.

### Использование скриптов запуска

В корневой директории проекта доступны скрипты start_backend.bat и start_frontend.bat для быстрого запуска бэкенда и фронтенда соответственно в среде Windows.

## 10. Разработка и расширение

### Добавление новых эндпоинтов

Для добавления нового эндпоинта создайте функцию в соответствующем файле endpoints, определите Pydantic схемы для запроса и ответа в schemas, добавьте бизнес-логику в services и зарегистрируйте роут в router.

### Расширение AI функциональности

Для добавления новых AI функций расширьте класс AIService новыми методами, использующими OpenAI API или другие языковые модели. Определите Pydantic модели для структурирования результатов и добавьте соответствующие промпты в prompt.py.

### Многоязычность

Интерфейс и промпты написаны на русском языке. Для добавления поддержки других языков необходимо обновить тексты промптов в backend/app/core/prompt.py и backend/app/services/ai_service.py, а также добавить переводы интерфейса во фронтенде.

### Масштабирование

Для горизонтального масштабирования рекомендуется использовать контейнеризацию с Docker и оркестрацию с Kubernetes. Векторные хранилища могут быть вынесены в отдельное хранилище (например, Pinecone или Weaviate) для поддержки множества инстансов бэкенда.
